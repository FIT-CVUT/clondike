47 /** @defgroup tcmi_ctlfs_file_class tcmi_ctlfs_file class
48  *
49  * @ingroup tcmi_ctlfs_entry_class
50  *
51  * A file in the TCMI control file system is used to export object
52  * interfaces. An object or a component creates a new control file
53  * for each method it wants to made available to the user space. The
54  * design idea is an extension of how linux sysctl works.
55  *
56  * - When a user space application reads from a file, the associated
57  * object is notified by calling its registered method. The result is
58  * communicated back to user space. Similarly to sysctl, there are
59  * various data types that can be passed to the objects method. Their
60  * conversion from/to ascii is handled by special handlers.
61  *
62  * - When a user space program writes data to the file, the data is
63  * parsed by the TCMI file object and the binary form is passed on to
64  * the registered object's method.
65  *
66  * As in sysctl, following functions for parsing ascii input data are provided:
67  * - string
68  * - an array of integers
69  * - in addition, a raw data can be passed into the method of the registered object.
70  *
71  * Based on this we can create three types of files:
72  * - integer files - methods of file instantiators accept/return integer data.
73  * Conversion from/to ascii is handled internally by this class
74  * - string files - methods of file instantiators accept/return string data.
75  * String parsing (overflow checks, new lines conversion) is
76  * handled internally by this class.
77  * - raw files - methods of file instantiators accept raw data.
78  * No conversion is performed.
79  *
80  * The object that creates the control file, specifies maximum allowed
81  * data length as it is in sysctl.
82  *
83  * To prevent code duplication, this class reuses the procfs
84  * functions(sysctl.c) - proc_dostring(), proc_dointvec() for ascii/binary
85  * conversions.
86  *
87  * When the data is being passed to registered object's method, a
88  * mutual access guarantees integrity (implemented using mutex)
89  *
90  *
91  * @{
92  */
93 
94 /** Describes the type of the file unregistration that is taking
95  * place.
96  *
97  * - regular way - is explicit file unregistration without using the
98  * file. This is requires locking the actual file, before
99  * unregistering its object.
100  * - from a method of the registered object - this is a special
101  * situation when the file is already locked (access is serialized by
102  * read/write file operations). Therefore, no locking is needed.
103  */

169 /**
170  * <<public>> Sets the unregistered flag of the file. This is used when the
171  * associated object is about to unregister from the file. However,
172  * the object needs to be guaranteed that from now on, no new process

174  * unregistered flag does.
