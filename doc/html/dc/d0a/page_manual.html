Overview

Shell scripts in scripts/ directory provide a comfortable way to accomplish basic setup and migration procedures. Full control over TCMI is provided by the TCMI ctlfs entries provided by CCN and PEN resp. In following examples node hpecko will be acting as CCN and node clare will be acting as PEN.

Loading/Unloading

The whole framework is loaded via load.sh script. After successful startup, there will be ctlfs entries for PEN and CCN available in /clondike.

Shutdown of the framework is accomplished by the unload.sh shell script. The script succeeds only if there are no migrated processes.

Setting up Listening on CCN

On CCN, it is necessary to setup listening, so that PEN's are able to join the cluster. This is done by the listen.sh script: 

hpecko:# ./scripts/listen.sh tcp:192.168.0.5:54321
hpecko:# find /clondike/ccn/listening-on/
/clondike/ccn/listening-on/
/clondike/ccn/listening-on/listen-00
/clondike/ccn/listening-on/listen-00/archname
/clondike/ccn/listening-on/listen-00/peername
/clondike/ccn/listening-on/listen-00/sockname

Connecting PEN to CCN

A PEN is allowed to connect to multiple CCN's, this is done via connect.sh script that takes the same argument as listen.sh: 

clare:# ./scripts/connect.sh tcp:192.168.0.5:54321
clare:# find /clondike/pen/nodes/
/clondike/pen/nodes/
/clondike/pen/nodes/0
/clondike/pen/nodes/0/state
/clondike/pen/nodes/0/connections
/clondike/pen/nodes/0/connections/ctrlconn
/clondike/pen/nodes/0/connections/ctrlconn/archname
/clondike/pen/nodes/0/connections/ctrlconn/peername
/clondike/pen/nodes/0/connections/ctrlconn/sockname

In this example PEN 'clare' has connected to a node 192.168.0.5 on port 54321 - the CCN where we have previously setup the listening.

Migrating a Sample Process from CCN to PEN

Migrating a process from CCN to PEN and back requires having a user account clondike with uid=999 setup (part of the installation process). The home directory of the user has to be shared among both nodes via network filesystem. Further, the home mount has to yield the same pathname on both nodes.

Now, run the process under the clondike identity: 

capekj@clare:$ su - clondike
clondike@clare:~$ ./test_process out.txt &
[1] 12708
clondike@clare:~$ ps
PID TTY          TIME CMD
12704 pts/4   00:00:00 bash
20705 pts/4   00:00:00 test_process
21200 pts/4   00:00:00 ps

NOTE: currently, there is a limited support for restoring special files. A temporary workaround is to guarantee that the pseudo terminal device(a regular character) device associated with the process on CCN will be available and writable for user clondike on PEN too.

Now the process is ready for migration, we have PEN (node 0) connected to CCN: 

clare:# cat /clondike/ccn/nodes/0/connections/ctrlconn/peername
192.168.0.1:42861

The script emig.sh does the migration work: 

hpecko:# ./scripts/emig.sh 12708 0 ppm-p
Migrating process: 12708 to PEN: 0 method: ppm-p
hpecko:# find /clondike/ccn/mig/migproc/
/clondike/ccn/mig/migproc/
/clondike/ccn/mig/migproc/20705
/clondike/ccn/mig/migproc/20705/remote-pid
/clondike/ccn/mig/migproc/20705/migman

On PEN, we will observe the immigrated process: 

clare:# find /clondike/pen/mig/migproc/
/clondike/pen/mig/migproc/
/clondike/pen/mig/migproc/12779
/clondike/pen/mig/migproc/12779/remote-pid
/clondike/pen/mig/migproc/12779/migman
clare:# cat /clondike/pen/mig/migproc/12779/remote-pid
20705

Migrating a Sample Process home from PEN to CCN

Currently, the migration has to be initiated by the PEN. This is not a limitation of TCMI capabilities. The protocol currently implements only a testing subset of all migration and process control messages. From TCMI prospective there is no difference in the initiator.

clare:# ./scripts/mig-home.sh 12779
Migrating process:12779 home from pen
clare:# find /clondike/pen/mig/migproc/
/clondike/pen/mig/migproc/

hpecko:# find /clondike/ccn/mig/migproc/
/clondike/ccn/mig/migproc/
hpecko:# pmap 20705
20705:   ./test_process out.txt
08048000    396K r-x--  /test_process.20.12779.55654364
080ab000     12K rw---  /test_process.20.12779.55654364
080ae000    136K rw---  /test_process.20.12779.55654364
bffeb000      4K rw---    [ anon ]
bffec000     80K rw---  /test_process.20.12779.55654364
ffffe000      4K -----    [ anon ]
total      632K

As we can see the migrated process has disappeared from both - CCN and PEN ctlfs entries on both nodes. On CCN, the process address space is now fully mapped from the checkpoint image and the process is running.
