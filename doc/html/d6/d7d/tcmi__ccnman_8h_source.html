47 /** @defgroup tcmi_ccnman_class tcmi_ccnman class
48  *
49  * @ingroup tcmi_man_class
50  *
51  * This <<singleton>> class implements the CCN TCMI manager
52  * component. It is responsible for registration of new PEN's. Each
53  * migration manager is associated with one registered PEN. The user
54  * controls the cluster core node by manipulating \link tcmi_ctlfs.c
55  * TCMI control file system \endlink. The directory/file layout can be
56  * divided into three groups of files/directories based on their
57  * maintainers, which can be:
58  *
59  * - TCMI CCN manager
60  * - TCMI CCN migration manager
61  * - TCMI CCN preemptive migration component
62  * - TCMI CCN non-preemtive migration component
63  * - TCMI shadow process
64  *
65  *
66  \verbatim
67  ------------ T C M I m a n a g e r --------------
68  ccn/listen -> allows adding new interfaces where the CCN manager
69  will be listening on.
70  listening-on/1/ -> each listening interface has one associated directory
71  2/
72  .
73  .
74  5/iface -> file that contains the interface address
75 
76  stop-listen-all -> writing into it terminates listening on all interfaces
77 
78  stop-listen-one -> writing into it terminates specified listening
79 
80  stop -> writing into this stops all listenings, migrates all processes back
81  and terminates all migration managers.
82 
83  ----------- T C M I m i g r a t i o n m a n a g e r s --------------
84  nodes/1/ -> each registered PEN has a separate directory, maintained
85  by its migration manager that is in charge of it.
86  2/
87  .
88  .
89  9/connections/ -> all connections of a migration manager
90  ctrlconn/ -> control connection
91  arch -> contains architecture of the connection
92  localname -> local socket name (address)
93  peername -> remote socket name (address)
94  state - current state of the migration manager
95  load -> average load of the PEN, used for migration decisions
96 
97  ------------ T C M I s h a d o w p r o c e s s e s --------------
98  mig/migproc/1300/ -> similar to /proc, but contains PID's of migrated processes
99  one directory per process
100  2001/
101  .
102  .
103  6547/migman->../../nodes/7 -> a symbolic link pointing to
104  its current PEN
105  remote-pid -> process PID at its current PEN
106  ------------ T C M I p p m c o m p o n e n t --------------
107  emigrate-ppm-p -> writing a PID + PEN id pair into this file starts process
108  migration to the specified PEN.
109  migrate-home -> allows migrating a specified process back CCN
110  ------------ T C M I n p m c o m p o n e n t (not implemented) --------------
111  policy -> interface for the migration policy, the npm component
112  uses this interface to ask the migration policy for migration
113  decision.
114 
115  \endverbatim
116  *
117  * As has been mentioned, the TCMI CCN manager is responsible for PEN registration.
118  * Handling incoming connections and instantiation of new migration managers requires
119  * a separate execution thread.
120  *
121  * Generally, we assume a limited number of interfaces that the CCN
122  * will be listening on. The core datastructure for keeping track of
123  * listenings is \link tcmi_slotvec_class a slot vector \endlink.
124  *
125  *
126  *
127  * @{
128  */
