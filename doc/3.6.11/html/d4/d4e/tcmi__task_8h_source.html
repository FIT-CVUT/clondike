59 /** @defgroup tcmi_task_class tcmi_task class
60  *
61  * @ingroup tcmi_task_group
62  *
63  * TCMI task is an abstraction for any process in the system that
64  * participates in migration. This <<template>> class gathers
65  * common functionality for shadow tasks that represent a migrated
66  * process on CCN as well as for guest tasks that represent the
67  * migrated process on PEN.
68  *
69  * TCMI tasks are controlled from user space via migration component.
70  * Each task is associated with its migration manager. This ensures
71  * that all messages addressed to a particular task are also delivered
72  * to its message queue or transactions slotvector. A TCMI task is
73  * given a communication socket (KKC socket) upon its
74  * instantiation. It is the communication channel that it will use for
75  * sending messages out. Note, that message delivery is handled
76  * exclusively by the migration manager. This makes it very easy to
77  * move a task among TCMI migration managers.
78  *
79  * The main purpose of this class is execution of methods in the context
80  * of the shadow and guest process resp. These methods are submitted by the
81  * migration component into the method queue and include:
82  * - request to migrate a process to a PEN - includes PPM_P, PPM_V, NPM
83  * - request to migrate a process back to CCN - includes PPM_P, PPM_V, NPM
84  * - request to process any incoming TCMI messages from the message queue.
85  * - request to terminate
86  *
87  * Methods in the queue are processed via a 'method pump'. Each method
88  * returns a status that is used by the controlling component to make
89  * a decision what to do further. Status return codes have following
90  * meanings:
91  *
92  * - KEEP_PUMPING - the method pump should keep going as long as there
93  * are methods to process
94  *
95  * - MOVE_ME - stop pumping and tell the task controlling component to
96  * move it to a different migration manager (relevant for shadow tasks
97  * on CCN)
98  *
99  * - EXECVE_FAILED_KILL_ME - stop pumping and tell the task
100  * controlling component to kill this task. This status denotes the
101  * situation when the task tried issuing execve(typically when loading
102  * a new checkpoint or processing a non-preemptive migration request)
103  * and it failed. This status is needed as it also instructs the method
104  * pump not to discard the method wrapper as it has already been done
105  * by (tcmi_task_execve() method internally)
106  *
107  * - KILL_ME - stop pumping and tell the task controlling component to
108  * kill this task (issue do_exit()) and remove it from its migration
109  * manager
110  *
111  * - LET_ME_GO -stop pumping and tell the task controlling component
112  * leave the migration mode, leave the task as is.
113  *
114  * - REMOVE_AND_LET_ME_GO -stop pumping and tell the task controlling
115  * component to remove the task from its current migration manager,
116  * destroy the tcmi_task and and let the regular task go. For example,
117  * this status is returned by task_exit() method as it is called from
118  * the intercepted exit system call. Once the TCMI task is detached
119  * from its process the exit system call resumes and performs the
120  * actual exit.
121  *
122  * In addition, the task also holds the filename along with argv and
123  * envp arrays that will be executed. It is important to keep track of
124  * this data as they are required by the execve (e.g. when migrating a
125  * a task by executing it's checkpoint image). These resources
126  * eventually need to be released once the execve has been issued or
127  * when the task is destroyed. There is functionality that makes arguments
128  * and environment management easy.
129  *
130  * Abbreviations:
131  * - PPM_P - preemptive process migration using physical checkpoint image
132  * - PPM_V - preemptive process migration using virtual checkpoint image
133  * - NPM - non-preemptive process migration (using execve hook)
134  *
135  * @{
136  */

582 /**
583  * <<public>> Processes a specified signal. The work is delegated
584  * to the task specific method if one is specified. This allows the
585  * shadow and guest processes to handle the signal differently. In
586  * fact, they have to: - shadow process takes appropriate actions and
587  * forwards the signal to the guest using signal forwarding
588  * component(since the guest might not be in migration mode) - guest
589  * process takes only appropriate actions and is able to communicate
590  * with the shadow directly.
591  *
592  * @param *self - pointer to this task instance
593  * @param signr - signal that is to be processed
594  * @param *info - info for signal to be processed
595  *
596  * @return result of the task specific signal handling method or
597  * TCMI_TASK_KEEP_PUMPING
598  */
599 static inline int tcmi_task_do_signal(struct tcmi_task *self, unsigned long signr, siginfo_t *info)
600 {
601  int res = TCMI_TASK_KEEP_PUMPING;
602 
603  mdbg(INFO2, "Processing signal %lu", signr);
604 
605  if ( self->peer_lost && signr == SIGKILL ) {
606  mdbg(INFO2, "Peer task is dead, killing current task");
607  return TCMI_TASK_KILL_ME;
608  }
609
