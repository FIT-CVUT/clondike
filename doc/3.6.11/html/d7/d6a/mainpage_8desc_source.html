29  * \section sec_mainpage_intro Introduction
30  *
31  * Clondike is a project developed at FEE and FIT CTU, Prague. The main goal
32  * is to create a cluster of non-dedicated workstations. This web page
33  * is automatically generated from clondike sources and explains
34  * various implementation aspects. Currently, clondike has three main
35  * parts that are described in separate sections:
36  *
37  * - \link kkc_class Kernel-to-Kernel Communication library(KKC) \endlink
38  * - \link tcmi_module_class Task Checkpointing and Migration Infrastructure(TCMI) \endlink
39  * - \link proxyfs_module_class Proxy File System(ProxyFS) \endlink
40  * - Cache Clondike Filesystem - Support cache file system for measuring
41  *
42  * All clondike classes can be viewed in a <a href="modules.html">tree-like structure </a>

49  * \section sec_mainpage_style Software Design Style
50  *
51  * \subsection subsec_mainpage_style_overall Overall Considerations
52  *
53  * All components are designed in a strict object-oriented
54  * style. Since we are on a C-based platform, the OO style can't be
55  * enforced by the compiler - it has to be respected by the
56  * developer. There is total encapsulation of all objects, all
57  * accessor methods are implemented with \c static \c inline memory
58  * class. All optimization is done by the compiler. Each class resides
59  * in a separate module(header file + optional .c file). Declaration
60  * section that is meant to be private is wrapped by a preprocessor
61  * directive:
62  \verbatim
63  #ifdef CLASS_NAME_PRIVATE
64  .
65  . declaration of private methods
66  .
67  #endif
68  \endverbatim
69  * The class definition file then prior to including the header file
70  * defines this macro which makes the private declarations visible.
71  *
72  * All instance variables (even for singleton classes) are wrapped in
73  * a \c struct. Each regular instance method takes a pointer to this
74  * compound structure(denoted as \c self) as its first
75  * argument.
76  *
77  * Singleton instance methods of a class that doesn't use inheritance
78  * don't use \c self as argument. They access the instance directly.
79  *
80  * \subsection subsec_mainpage_style_ref Reference Counting
81  *
82  * We have to realize that most of the objects are being accessed by
83  * multiple threads of execution. This fact requires a solid instance
84  * creation/destruction scheme. There is no garbage collecting present
85  * in kernel. This solution uses reference counters - each object is
86  * responsible for its reference counting. Each thread operating on an
87  * object retains its own reference to it. Threads don't destroy the
88  * objects, they merely release the reference. When the last reference
89  * to an object has been released, the object then performs a \b self
90  * \b destruction. This approach provides a robust solution and is
91  * generally recommended by kernel developers. In fact, not having a
92  * proper reference counting in a multithreaded 'kernel application' is
93  * considered a bug.
94  *
95  * \subsection subsec_mainpage_style_synch Synchronization
96  *
97  * Speaking of multiple threads of execution, some objects require
98  * synchronization of threads accessing them. Linux kernel provides
99  * various synchronization primitives(spinlocks, semaphores,
100  * completions). Their use is specific to each individual situation
101  * and object. As a rule of thumb, the approach - '\em protect \em
102  * data \em not \em code' has been used.
103  *
104  * \subsection subsec_mainpage_style_inheritance Inheritance
105  *
106  * Inheritance is a very important 'software tool' that prevents code
107  * duplication and yields lighter and easier to maintain code. There
108  * is more than one way how it can be implemented in C. For this
109  * project a simple object composition has been chosen as the most
110  * efficient variant. A \a class_a that inherits from \a class_b
111  * contains an instance of \a class_b in its compound
112  * structure. Example:
113  \verbatim
114  struct class_b {
115  int b;
116  };
117  struct class_b* class_b_init(....);
118  \endverbatim
119  \verbatim
120  struct class_a {
121  struct class_b super; -> actual composition = inheritance
122  int a;
123  };
124  struct class_b* class_a_new(....);
125  \endverbatim
126 
127  * The \a class_b intentionally doesn't have a public constructor as
128  * it is considered abstract. The only public initializer, that it
129  * provides, is intended for the child classes to be called from their
130  * constructors.
131  *
132  * It is \b very important that \c super (always denoted like this
133  * throughout the source code) is placed at the very top of the \a
134  * class_a compound structure. With this layout, we can guarantee that
135  * a pointer to child instance also points to parent instance. Thus,
136  * the parent methods will treat it properly. The \a class_a_new
137  * constructor intentionally returns a pointer to the parent class
138  * object. This is to support polymorphism(see further).
139  *
140  * \subsection subsec_mainpage_style_polymorph Polymorphism & Template Design Patterns
141  *
142  * C is a language with type control, pure polymorphism is not
143  * possible. In order to have polymorphism we have to do inheritance
144  * (unless we want to get around with '\c void *' pointers only). The
145  * implementation takes a very efficient approach that is used
146  * everywhere in the kernel (e.g. VFS). The child class constructor
147  * supplies a vector of operations to the parent class and returns a
148  * pointer to the parent class instance. The parent class provides
149  * methods that are considered as the official object
150  * protocol. Instances of all classes that inherit from the same
151  * parent are thus considered \em polymorphic.
152  *
153  * For some objects we want the parent class instance to do some work
154  * prior and after calling the child method from the operations
155  * vector. E.g. a \link tcmi_msg_class TCMI migration message \endlink
156  * sends its header and delegates sending the body to the child class
157  * that knows how to do it. The \em template pattern can thus be
158  * considered as a natural extension of \em polymorphism. For example
159  * implementation see e.g. \link tcmi_msg_class tcmi_msg \endlink and
160  * \link tcmi_skel_msg_class tcmi_skel_msg \endlink.
161  *
162  * \subsection subsec_mainpage_style_coding Coding Style
163  *
164  * The coding style follows recommendations for linux kernel including
165  * K&R indentation. There are a few details that should be clearly
166  * stated here: Since there is no software exception/error handling
167  * mechanism(like try/catch or setjmp/longjmp statements), we have to
168  * deal with errors manually. This is a common approach used
169  * throughout the kernel. Each method (considering a non-trivial one)
170  * has two parts: - executive part (at the top, the actual
171  * functionality/work to be done) - error handling part (at the
172  * bottom)
173  *
174  * When an error is detected a \c goto is issued to a corresponding
175  * handler at the bottom. The handler then unrolls all actions that
176  * have executed so far(if possible) and returns an error code. The
177  * error code is generally a negative number. For methods that return
178  * a pointer, NULL signals an error. A typical example method that
179  * implements this error handling is any constructor - class method.
180  * See e.g. tcmi_shadowtask.c::tcmi_shadowtask_new().
181  *
182  * All class identifiers strictly have following prefix:
183  *
184  * - \c tcmi_ for \link tcmi_module_class TCMI \endlink
185  * - \c proxyfs_ for \link proxyfs_module_class ProxyFS \endlink
186  * - \c kkc_ for \link kkc_class KKC \endlink
187  * - \c ccfs_ for CCFS
188  *
189  * based on the subsystem they belong to. The implementation of a
190  * class is carried out in the .h and .c file of the same name as the
191  * class is.
192  *
193  * Each method name carries the identifier of the class that it
194  * belongs to. This rather tedious(but consistent) coding decision
195  * increases code
196  * readibility. E.g. tcmi_task.c::tcmi_task_set_local_pid() represents
197  * a local PID setter of \link tcmi_task_class tcmi_task\endlink.
198  *
199  * \section sec_mainpage_patch Linux Kernel Patch
200  *
201  * Using \ref tcmi_module_class with Linux kernel required a few
202  * modifications in the kernel itself. This section describes
203  * necessary changes.
204  *
205  * \subsection subsec_mainpage_patch_hooks TCMI hooks (include/clondike/tcmi/tcmi_hooks.h)
206  *
207  * This is an extension that provides an easy way to install hooks
208  * anywhere in the Linux kernel. A hook then provides a mechanism for
209  * external external subsystems register their functionality. Having
210  * the system setup like this allows having the entire TCMI outside
211  * the kernel.
212  *
213  * Creating a new hook requires:
214  * - declaring the hook in the \a tcmi_hooks.h header
215  * - adding the hook call at the desired place in the kernel
216  *
217  * E.g. the \ref subsec_mainpage_patch_sig_unused "SIGUNUSED" is
218  * installed in \a kernel/signal.c as follows:
219  \verbatim
220  .
221  .
222  if (signr == SIGUNUSED) {
223  spin_unlock_irq(&current->sighand->siglock);
224  TCMI_HOOKS_CALL(sig_unused, regs);
225  goto relock;
226  }
227  .
228  .
229  \endverbatim
230  *
231  * \subsection subsec_mainpage_patch_sig_unused SIGUNUSED/SIGSYS (kernel/signal.c)
232  *
233  * TCMI hook - the default handler for this signal (31) - has been
234  * installed into \a get_signal_to_deliver() function. In addition,
235  * the signal has been added among \em kernel \em signals. This
236  * prevents any user processes (effectively forbids) from setting
237  * their own signal handler for this signal.
238  *
239  * \subsection subsec_mainpage_patch_task_struct task_struct (include/linux/sched.h)
240  *
241  * Shadow and guest processes require an entry in process
242  * descriptor. The entry carries:
243  * - pointer to the TCMI task (type void*, so that the actual \link
244  * tcmi_task_class tcmi_task \endlink doesn't need to be part of
245  * kernel sources).
246  * - pointer to the migration mode handler
247  * - pointer to data available for the handler (currently unused)
248  *
249  * \subsection subsec_mainpage_patch_do_exit do_exit (kernel/exit.c)
250  *
251  * TCMI hook for this system call is installed, so that the framework
252  * is able to keep track of terminating migrated(\link
253  * tcmi_guesttask_class guest \endlink) processes.
254  *
255  * \subsection subsec_mainpage_patch_tmp_debug Debugging (clondike/tcmi/tcmi_dbg.h, arch/i386/fault.c, mm/memory.c)
256  *
257  * There are debug messages directly in the kernel that will show up
258  * in the kernel debug log only for tcmi_tasks. Currently, there are
259  * debug messages in page fault handler to verify correct checkpoint
260  * restart. This will be removed in the future.
261  *
262  * \subsection subsec_mainpage_patch_kconfig Kconfig(Makefile, arch/i386/Kconfig, clondike/tcmi/Kconfig)
263  *
264  * The kernel configuration has been extended. This allows enabling
265  * TCMI hooks in the kernel. Also, support for CCN and PEN on the same
266  * node can be configured.
